Le produit

Tout se passe sur un seul écran.

  • Le lecteur — vidéo, timeline à marqueurs, contrôles.
  • Le calque d'annotation — on dessine directement sur l'image.

Sujet A · ce que le sujet exige

Diffuser une vidéo que personne ne peut piller.

✓ Diffusion HLS (playlist + segments) exigé
La vidéo est servie en HLS : un manifeste index.m3u8 + des segments .ts, jamais un fichier unique téléchargeable.
✓ Flux chiffré AES-128 exigé
À l'ingestion, ffmpeg chiffre chaque segment en AES-128 (-hls_key_info_file) avec une clé + IV propres au contenu.
✓ Clé délivrée seulement sur token valide — refus par défaut exigé
GET /keys/:id gardé par le JWT + l'ACL. Sans token, expiré, mauvais tenant ou clé révoquée → 401 / 403 / 404. C'est le serveur de clés éphémères.
✓ Déploiement local en une commande exigé
docker compose up lance core + frontend + hls + engine. Aucune étape manuelle obscure.
✓ Sans cloud payant exigé
Tout tourne en local (Docker). Aucune dépendance à un service payant.
✓ Modèle de menace assumé évalué
Quoi protéger, contre quoi, hypothèses et limites — détaillé en fin de présentation (+ docs/P2-threat-model.md).
Sujet A · le cœur

Le serveur de clés éphémères.

La clé ne sort jamais sans une décision explicite. Chaque requête est re-vérifiée de zéro.

  • Refus par défaut — on n'autorise qu'après avoir tout validé.
  • Isolation tenant — mauvais companyId → 404, on ne révèle rien.
  • Révocation — clé retirée → 403 immédiat, réversible.
  • Audit — chaque accès (accordé/refusé) est journalisé.
keys.service.ts
// GET /keys/:contentId — refus par défaut async getKey(id, user, ip) { const c = this.contents.find(id) if (!c || c.companyId !== user.companyId) throw NotFound // autre tenant → 404 if (!this.contents.isAllowed(id, user)) throw Forbidden // pas d'accès → 403 if (c.revoked) throw Forbidden // clé révoquée → 403 this.audit(id, user, ip, 'granted') return readKey(id) // ✓ clé AES-128 }
Sujet A · la preuve demandée

Avec token, ça lit. Sans, la clé est refusée.

# 1) Sans token → refus par défaut $ curl -s -o /dev/null -w "%{http_code}" /keys/poc 401 Unauthorized # 2) Bon utilisateur, bonne entreprise → clé délivrée $ curl -H "Authorization: Bearer $ALICE" /keys/poc 200 ····clé AES-128 (binaire)···· # 3) Utilisateur d'une AUTRE entreprise → on ne révèle rien $ curl -H "Authorization: Bearer $EVE" /keys/poc 404 Contenu introuvable # 4) Admin révoque la clé, en direct → lecture coupée $ curl -X POST -H "Authorization: Bearer $ADMIN" /admin/contents/poc/revoke $ curl -H "Authorization: Bearer $ALICE" /keys/poc 403 Accès refusé pour ce contenu
Sujet B · ce que le sujet exige

Détecter et bloquer les abus en temps réel.

✓ Détection temps réel + réaction visible exigé
Le trafic est analysé à chaque requête ; au-delà des seuils, la réaction est un 403 réel (le segment est refusé), pas un simple log.
✓ Partage de compte à grande échelle signal
Multi-session : un même compte vu depuis > 3 IP distinctes en 5 min → alerte.
✓ Accès via VPN / proxies signal
Liste de réseaux (CIDR) de réputation : une IP qui matche est signalée (proxy_ip).
✓ Débit d'aspiration (scraping) signal
Segments .ts par acteur en fenêtre glissante 60 s : > 8 → alerte, > 20 → blocage.
✓ Démo avec scripts d'attaque (avant / après) exigé
Un script qui aspire les segments passe d'abord, puis se fait couper net dès le seuil franchi (via nginx auth_request).
✓ Règles documentées, limites assumées évalué
Seuils et faux positifs discutés dans docs/P2-anti-scraping.md.
security.service.ts
// fenêtre glissante, par acteur (compte|IP) const n = this.segmentsFor(actor, 60_000) if (n > 20) return { blocked:true } // → 403 if (n > 8) this.alert('segment_scrape')
# script d'aspiration $ for i in $(seq 1 40); do curl .../seg$i.ts; done seg1..seg20 200 # l'abus passe… seg21+ 403 # …puis bloqué net
Sujet B · détection & blocage

Le blocage est réel.

nginx interroge le Core en auth_request avant chaque segment : un 403 du Core = segment refusé par nginx.

  • Rate-limit — 100 req / 60 s / IP en amont.
  • Bans IP — blocage manuel depuis le dashboard, persisté.
  • Dashboard live — alertes horodatées, MAJ toutes les 2 s.
  • Anti-usurpation — X-Forwarded-For n'est cru que des proxies de confiance.

Sujet A · ce qui rend la vidéo trouvable

Recherche sémantique et sous-titres multilingues.

Recherche sémantique

  • Embeddings MiniLM multilingue sur chaque segment
  • Requête naturelle → segments les plus proches (cosinus)
  • « où on parle de sécurité ? » → saut au bon timecode
  • Exposée au Core : POST /contents/:id/search

Sous-titres multilingues

  • NLLB-200 traduit le transcript segment par segment
  • fr · en · es · ar par défaut, extensible (jusqu'à 200 langues)
  • Incrémental : seules les langues manquantes sont traduites
  • Incrusté dans le lecteur (bouton CC de la View)

La transcription est le minimum vital. La valeur, c'est tout ce qu'on en tire au-dessus : chercher, traduire, résumer, chapitrer.
